# Shore Trace 决策记录
本文件只记录长期有效、容易反复讨论的产品和技术决策。临时计划、进度和待办写入 `docs/HANDOFF.md`。
## ADR-001：第一版使用 localStorage
状态：已采纳
### 决策
第一版将学习记录保存在当前浏览器 `localStorage` 中，并通过 `studyRecordRepository` 隔离存储实现。

### 原因
当前是个人、本地优先 MVP，目标是验证记录闭环和连续使用负担，暂不需要登录、后端、云同步或多设备同步。

### 代价
数据只存在于当前浏览器；清理数据、切换浏览器或切换设备后不会自动同步，容量和可靠性也有限。

### 重新评估条件
当需要多设备同步、复杂查询、数据规模明显增长或更强恢复能力时，再评估 IndexedDB 或后端。

## ADR-002：第一版不使用 Pinia
状态：已采纳
### 决策
第一版不引入 Pinia，页面状态和业务流程优先使用 Composition API 与现有 composables。

### 原因
当前页面和共享状态规模有限，Composition API 足够，能避免增加依赖、模板代码和状态层复杂度。

### 代价
跨页面共享状态增长后，composables 可能更难拆分、追踪和调试。

### 重新评估条件
当多个页面频繁共享可变状态，或状态流转难以定位时，再评估 Pinia。

## ADR-003：模块按需保存
状态：已采纳

### 决策
未学习模块不创建空对象；保存前过滤空模块；空模块不进入历史和统计。

### 原因
按需保存能减少数据污染和表单负担，让历史与统计只反映真实输入过的学习内容。

### 代价
无法仅凭存储数据区分“主动选择后未填写”和“从未选择过”的模块。

### 重新评估条件
当需要分析计划完成率、主动跳过模块或未填写原因时，再讨论是否保存显式空状态。

## ADR-004：统计只基于高可信数据
状态：已采纳

### 决策
统计优先基于学习时长、做题数、错题数和固定错因；不统计缺乏可靠计时来源的平均每题用时；不基于不足样本生成强结论。

### 原因
第一版数据来自手动记录，可信度和连续性需要先验证。事实型统计比推断型结论更稳定。

### 代价
统计表达会更克制，不提供复杂效率分析、趋势预测或训练建议。

### 重新评估条件
当记录连续性稳定、样本量足够、计时来源更可靠，且用户需要深入分析时，再扩展统计口径。

## ADR-005：第一版不做 AI 复盘
状态：已采纳

### 决策
第一版不提供 AI 复盘，只提供基于结构化数据的事实型观察。

### 原因
应先验证记录数据是否稳定、连续、可信。AI 复盘过早引入会扩大 MVP 范围并增加误导风险。

### 代价
产品不会自动生成个性化训练建议或复盘文本。

### 重新评估条件
当连续记录形成足够历史数据，且事实型观察已不能满足复盘需求时，再评估 AI 能力。

## ADR-006：页面不直接访问 localStorage
状态：已采纳

### 决策
所有数据读写通过 `src/repositories/studyRecordRepository.js`，页面和 composables 只消费 repository 返回结果。

### 原因
集中存储边界便于处理异常本地数据、保存失败、导出、未来迁移和兼容旧数据。

### 代价
简单读写也需要经过 repository，局部实现会多一层调用。

### 重新评估条件
除非底层存储架构整体迁移，否则不放开页面直接访问存储。

## ADR-007：设备本地日期作为业务日期
状态：已采纳

### 决策
记录日期使用设备本地 `YYYY-MM-DD`；`createdAt` 和 `updatedAt` 可以使用 ISO 8601；禁止用 `toISOString().slice(0, 10)` 生成业务日期。

### 原因
学习记录按用户本地自然日组织，UTC 日期会在时区边界导致日期偏移。

### 代价
跨时区迁移或旅行场景下，历史记录仍以记录当时设备本地日期为准。

### 重新评估条件
当引入账号、云同步或跨时区多设备协作时，再讨论日期模型和时区策略。

## ADR-008：暂时保留自定义错因，但不继续扩展
状态：已采纳

### 决策
保留已经实现的自定义错因记录能力，但不增加排行、筛选、同义词归并或复杂统计。

### 原因
自定义错因能补充固定标签表达不足，但自由文本噪音较高，不适合作为第一版复杂统计依据。

### 代价
自定义错因主要作为记录补充，统计页不会完整反映这些文本信息。

### 重新评估条件
如果实际使用率低，可评估移除；如果使用率高且文本质量稳定，再评估轻量归类或统计。

## ADR-009：本地数据读取采用前向兼容保留策略
状态：已采纳

### 决策
当 `localStorage` 顶层数据结构有效且版本匹配时，当前版本读取记录时忽略未知模块 key，但保存和导出时尽量保留未知模块、已知模块内部未知字段、记录级未知字段和顶层未知字段。非法 JSON、顶层为 `null`/数组/字符串、版本不匹配或 `records` 结构异常仍然阻止保存覆盖。

### 原因
历史版本或未来版本可能写入当前版本不认识的字段。读取时直接让全库失效会降低可用性；保存时直接丢弃未知字段又可能造成不可恢复的数据丢失。因此采用“已知数据可用、未知数据保留、异常顶层不覆盖”的折中策略。

### 代价
导出的 JSON 可能包含当前 UI 不展示的字段；repository 保存逻辑比简单整包覆盖更复杂，需要测试保护。

### 重新评估条件
当实现数据导入、版本迁移、云同步或 schema migration 时，应重新设计更明确的迁移和冲突处理策略。
